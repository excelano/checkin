# CheckIn: Potential Features

Ideas under consideration, not yet committed to the build. Each entry notes the rough approach and the main tradeoff so we can judge priority later. Shipped functionality lives in FEATURES.md.

## Widget self-refresh (fetch its own data)

The medium widget currently displays a snapshot the app writes, so it only shows fresh data after the app refreshes, whether in the foreground or via background app refresh. The widget extension could instead fetch its display data from Microsoft Graph directly in its timeline provider, keeping it current without depending on the app running. The extension already authenticates and calls Graph for the presence toggle, so the capability is proven. The cost is that the calendar, mail, and chat fetches live only in the app's GraphClient and Inbox today, so this means extracting a shared fetcher or duplicating that logic in the extension, along with more battery and network use. Timeline reloads stay under the OS's control, so this delivers fresh data on each system reload of a visible widget rather than literally on swipe. It is the closest achievable to "current whenever I look at it" without opening the app.

## Auto-offline outside working hours

Read the user's Outlook working hours from Microsoft Graph (the `workingHours` object on `/me/mailboxSettings`, which carries the working days, start and end times, and time zone) and automatically set presence to Offline outside those hours, releasing it during working hours. This would run on the background refresh cycle and complement the manual availability toggle: the toggle is an explicit "I am here" or "I am busy" override, while this is a passive "stop showing me as around after hours." It should be opt-in. Setting Offline uses `setUserPreferredPresence` (availability Offline, activity OffWork) and is still gated by an active presence session, the same constraint that governs the toggle. Because it overlaps with the manual toggle, it needs a clear precedence rule, most likely that a manual override wins until it expires or the user clears it. The privacy posture is unchanged, since it reads and writes only Microsoft Graph on the authenticated device.

## Apple Watch Tier 2: independent auth on the wrist

The shipped watch app is Tier 1: read-only, fed non-credential status data the phone pushes over WatchConnectivity, holding no token and making no Graph call. Tier 2 is an opt-in setting (default off) that lets the watch authenticate itself, hold its own device-bound token, and fetch fresh data and perform actions while standalone on cellular out of range of the phone. Even with it on, the phone stays the primary data path when in range; the watch self-fetches only when it cannot reach the phone, and for actions. The privacy line that must hold is that the watch's token is its own, acquired by its own sign-in and stored in the watch's own keychain with iCloud Keychain sync off; the phone's token is never copied to the watch and the watch never gets the phone's keychain access group. Implementation links CheckInGraph into the watch and gives it its own MSAL client and its own Entra redirect URI under the existing app registration.

The gate, and the reason this stays an idea rather than a commitment, is a spike. Microsoft sign-in on watchOS via `ASWebAuthenticationSession` (a full web login with MFA on the watch screen) is feasible on paper but unproven on this hardware, and even if it works, enterprise Conditional Access can reject the watch as a non-compliant device per tenant. So the tradeoff is real engineering and auth-surface risk for a capability most users will not need, and the feature must fall back silently to Tier 1 whenever the spike, the runtime sign-in, or a tenant blocks it. Prove the spike on-device before building anything else. David likes the app as it is and sees no current need, so this is parked, not scheduled.
