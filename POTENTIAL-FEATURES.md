# CheckIn: Potential Features

Ideas under consideration, not yet committed to the build. Each entry notes the rough approach and the main tradeoff so we can judge priority later. Shipped functionality lives in FEATURES.md.

## Auto-offline outside working hours

Read the user's Outlook working hours from Microsoft Graph (the `workingHours` object on `/me/mailboxSettings`, which carries the working days, start and end times, and time zone) and automatically set presence to Offline outside those hours, releasing it during working hours. This would run on the background refresh cycle and complement the manual availability toggle: the toggle is an explicit "I am here" or "I am busy" override, while this is a passive "stop showing me as around after hours." It should be opt-in. Setting Offline uses `setUserPreferredPresence` (availability Offline, activity OffWork) and is still gated by an active presence session, the same constraint that governs the toggle. Because it overlaps with the manual toggle, it needs a clear precedence rule, most likely that a manual override wins until it expires or the user clears it. The privacy posture is unchanged, since it reads and writes only Microsoft Graph on the authenticated device.

## Teams chats: show the thread back to your last reply

Status: promoted to active build 2026-06-03, targeting the next release alongside the mark-unread spinner fix. Design is final (below). Moves to FEATURES.md when it ships.

Today a Teams chat shows only the single most recent message, in the row and in the preview sheet, which loses the context of a back-and-forth that arrived since you last looked and makes it hard to reply intelligently without leaving the app. The change lives entirely in the chat preview sheet (the row stays a one-line preview). When you tap a chat, the sheet automatically shows the run of messages back to the last one you sent, with your own message anchoring the top, the replies you have not answered below it, and the newest nearest the Reply box, so you can read what you missed and reply in context. No extra tap: the context loads for free on open.

The fetch reads `/chats/{chatId}/messages` once, sorts newest-first client-side (chat messages do not guarantee server ordering), and walks down collecting messages until it reaches the first one from the signed-in user or hits a cap of 20, whichever comes first. System event messages (joins, renames) are skipped and HTML bodies stripped, reusing the helpers `pendingChats` already uses. When you have never posted in the thread, the walk reaches the start and shows the whole thread, still bounded by the cap. There is no in-app paging: if there is more history than the cap, the sheet shows a visible "Open in Teams" button that deep-links to the exact chat via the Graph-supplied `webUrl`, handing the full archive off to Teams rather than reimplementing a chat client.

Two refinements keep it feeling instant and resilient. The sheet seeds with the last message we already hold from the summary fetch (`chat.preview`) and renders it immediately, then loads the earlier context in above it with a small inline spinner, so the sheet never shows a blank wait. If the messages fetch fails, or `webUrl` and messages are unavailable, the sheet degrades to exactly today's behavior: the last-message line plus Reply and Open in Teams. The "Open in Teams" affordance is also the fix for a discoverability gap, since today that action is buried in a long-press context menu most users never find.

Privacy posture is unchanged: it reads only Microsoft Graph on the authenticated device. Confirm before building that reading full chat message bodies needs no scope beyond the `Chat.Read` we already hold for previews and members. The deep-link reliability caveat from the meeting-join path applies: the `https://teams.microsoft.com/l/chat/...` universal link lands in the Teams app only when Teams is installed and claims it, otherwise it opens Teams web in Safari silently, and `webUrl` is nil on some legacy chats and tenants, so the handoff degrades to generic Teams in those cases. Verify the actual landing behavior and how often `webUrl` is populated on-device before relying on it.

## Apple Watch Tier 2: independent auth on the wrist

The shipped watch app is Tier 1: read-only, fed non-credential status data the phone pushes over WatchConnectivity, holding no token and making no Graph call. Tier 2 is an opt-in setting (default off) that lets the watch authenticate itself, hold its own device-bound token, and fetch fresh data and perform actions while standalone on cellular out of range of the phone. Even with it on, the phone stays the primary data path when in range; the watch self-fetches only when it cannot reach the phone, and for actions. The privacy line that must hold is that the watch's token is its own, acquired by its own sign-in and stored in the watch's own keychain with iCloud Keychain sync off; the phone's token is never copied to the watch and the watch never gets the phone's keychain access group. Implementation links CheckInGraph into the watch and gives it its own MSAL client and its own Entra redirect URI under the existing app registration.

The gate, and the reason this stays an idea rather than a commitment, is a spike. Microsoft sign-in on watchOS via `ASWebAuthenticationSession` (a full web login with MFA on the watch screen) is feasible on paper but unproven on this hardware, and even if it works, enterprise Conditional Access can reject the watch as a non-compliant device per tenant. So the tradeoff is real engineering and auth-surface risk for a capability most users will not need, and the feature must fall back silently to Tier 1 whenever the spike, the runtime sign-in, or a tenant blocks it. Prove the spike on-device before building anything else. David likes the app as it is and sees no current need, so this is parked, not scheduled.
